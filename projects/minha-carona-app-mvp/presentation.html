DynamoDB Single Table Design

The application uses a single DynamoDB table with a well-designed key structure to support all access patterns efficiently.

Table Structure

PK (Partition Key) SK (Sort Key) Description
USER#<id> METADATA User registration (Resident/Driver)
CARONA#<id> DETALHES Ride details and metadata
CARONA#<id> RESERVA#<user_id> Reservation linking passenger to ride
CONDOMINIO#<id> CARONA#<data>#<id> Index for admin to list all rides

Global Secondary Index (GSI1)

GSI1PK GSI1SK Use Case
STATUS#<status> DATA_HORA (ISO format) List all open rides for today, sorted by time

Example: GSI1PK = STATUS#ABERTA returns all open rides ordered by date/time for instant listing.

gRPC: High-Performance Internal Communication

The application implements a hybrid architecture using gRPC for internal microservice communication where performance and low latency are critical, while maintaining REST APIs for public-facing endpoints.

Why gRPC?

  • Performance: 5-10ms latency vs 50-100ms REST
  • Protocol Buffers: Binary serialization (smaller payloads)
  • HTTP/2: Multiplexing and header compression
  • Streaming: Bidirectional real-time updates
  • Type Safety: Strongly typed contracts

When to Use gRPC

  • ‚úÖ Internal microservice communication
  • ‚úÖ High-frequency operations (checking availability)
  • ‚úÖ Low-latency requirements
  • ‚úÖ Streaming large datasets
  • ‚úÖ Synchronous critical operations

When NOT to Use

  • ‚ùå Public APIs (frontend/mobile)
  • ‚ùå External service integration
  • ‚ùå Simple CRUD operations
  • ‚ùå Browser-based clients

Key Benefits of gRPC in This Architecture

  • Reduced Latency: 10x faster than REST for internal calls
  • Lower Bandwidth: Protocol Buffers reduce payload size by 50-80%
  • Better Scalability: HTTP/2 multiplexing handles more concurrent requests
  • Type Safety: Compile-time validation prevents runtime errors
  • Streaming Support: Real-time updates without polling
  • Language Agnostic: Works across Python, Go, Java, Node.js, etc.

Reservation Flow (SQS Pattern)

1. User Request

  • Resident clicks "Reserve" button
  • Frontend sends request to API
  • API validates request

2. Queue Message

  • API sends message to SQS
  • Message contains: carona_id, passenger_id, driver_id
  • Queue ensures sequential processing

3. Worker Processing

  • Lambda worker processes queue
  • Atomically checks available seats
  • Creates reservation or rejects

4. Notification

  • SNS sends confirmation
  • Driver and passenger notified
  • Status updated in DynamoDB

Why SQS?

SQS prevents race conditions when multiple users try to reserve the last available seat simultaneously. The queue ensures atomic, sequential processing, eliminating duplicate reservations and maintaining data consistency.
